# 代理

配置和使用专门的代理。

代理是专用的 AI 助手，可以配置用于特定任务和工作流程。它们允许你创建具有自定义提示、模型和工具访问权限的专注工具。

**提示**

使用 plan 代理来分析代码并审查建议，而无需进行任何代码更改。

你可以在会话期间切换代理，或使用 `@` 提及来调用它们。

---

## 类型

OpenCode 中有两种类型的代理：主代理和子代理。

---

### 主代理

主代理是你直接交互的主要助手。你可以使用 **Tab** 键或配置的 `switch_agent` 快捷键在它们之间切换。这些代理处理你的主要对话。工具访问通过权限配置 — 例如，Build 具有所有工具启用，而 Plan 受限。

**提示**

你可以在会话期间使用 **Tab** 键在主代理之间切换。

OpenCode 提供两个内置的主代理：**Build** 和 **Plan**。我们将在下面查看这些。

---

### 子代理

子代理是主代理可以调用以执行特定任务的专门助手。你也可以通过在消息中**@ 提及**它们来手动调用它们。

OpenCode 提供两个内置的子代理：**General** 和 **Explore**。我们将在下面查看这些。

---

## 内置代理

OpenCode 提供两个内置的主代理和两个内置的子代理。

---

### 使用 build

*模式*：`primary`

Build 是**默认**的主代理，具有所有工具启用。这是需要完整文件操作和系统命令的开发工作的标准代理。

---

### 使用 plan

*模式*：`primary`

一个受限的代理，设计用于规划和分析。我们使用权限系统给你更多控制并防止意外更改。默认情况下，以下所有内容都设置为 `ask`：

- `file edits`：所有写入、补丁和编辑
- `bash`：所有 bash 命令

此代理在你想让 LLM 分析代码、建议更改或创建计划而不实际修改代码库时很有用。

---

### 使用 general

*模式*：`subagent`

一个通用代理，用于研究复杂问题和执行多步骤任务。具有完整的工具访问权限（除 todo 外），因此在需要时可以修改文件。使用此工具并行运行多个工作单元。

---

### 使用 explore

*模式*：`subagent`

一个快速、只读代理，用于探索代码库。无法修改文件。当你需要按模式快速查找文件、搜索代码中的关键字或回答关于代码库的问题时使用此工具。

---

### 使用 compaction

*模式*：`primary`

隐藏的系统代理，用于将长上下文压缩为较小的摘要。它在需要时自动运行，并且在 UI 中不可选择。

---

### 使用 title

*模式*：`primary`

隐藏的系统代理，用于生成简短的会话标题。它自动运行，并且在 UI 中不可选择。

---

### 使用 summary

*模式*：`primary`

隐藏的系统代理，用于创建会话摘要。它自动运行，并且在 UI 中不可选择。

---

## 使用

1.  对于主代理，在会话期间使用 **Tab** 键在它们之间切换。你也可以使用配置的 `switch_agent` 快捷键。

2.  子代理可以被调用：
    - **自动**由主代理根据其描述执行专门任务。
    - **手动**通过在消息中**@ 提及**子代理。例如。

        ```
        @general help me search for this function
        ```

3.  **会话之间的导航**：当子代理创建子会话时，使用 `session_child_first`（默认：**<Leader>+Down**）从父会话进入第一个子会话。

4.  一旦进入子会话，使用：
    - `session_child_cycle`（默认：**Right**）切换到下一个子会话
    - `session_child_cycle_reverse`（默认：**Left**）切换到上一个子会话
    - `session_parent`（默认：**Up**）返回父会话

    这允许你在主对话和专门的子代理工作之间切换。

## 配置

你可以自定义内置代理或通过配置创建自己的代理。代理可以以两种方式配置：

---

### JSON

在你的 `opencode.json` 配置文件中配置代理：

opencode.json

```json
{
  "$schema": "https://opencode.ai/config.json",
  "agent": {
    "build": {
      "mode": "primary",
      "model": "anthropic/claude-sonnet-4-20250514",
      "prompt": "{file:./prompts/build.txt}",
      "tools": {
        "write": true,
        "edit": true,
        "bash": true
      }
    },
    "plan": {
      "mode": "primary",
      "model": "anthropic/claude-haiku-4-20250514",
      "tools": {
        "write": false,
        "edit": false,
        "bash": false
      }
    },
    "code-reviewer": {
      "description": "Reviews code for best practices and potential issues",
      "mode": "subagent",
      "model": "anthropic/claude-sonnet-4-20250514",
      "prompt": "You are a code reviewer. Focus on security, performance, and maintainability.",
      "tools": {
        "write": false,
        "edit": false
      }
    }
  }
}
```

---

### Markdown

你也可以使用 markdown 文件定义代理。将它们放在：

- 全局：`~/.config/opencode/agents/`
- 项目级：`.opencode/agents/`

~/.config/opencode/agents/review.md

```markdown
---description: Reviews code for quality and best practicesmode: subagentmodel: anthropic/claude-sonnet-4-20250514temperature: 0.1tools:  write: false  edit: false  bash: false---
You are in code review mode. Focus on:
- Code quality and best practices- Potential bugs and edge cases- Performance implications- Security considerations
Provide constructive feedback without making direct changes.
```

markdown 文件名成为代理名称。例如，`review.md` 创建一个 `review` 代理。

## 选项

让我们详细看看这些配置选项。

---

### description

使用 `description` 选项提供工具及其使用时机的简短描述。

opencode.json

```json
{
  "agent": {
    "review": {
      "description": "Reviews code for best practices and potential issues"
    }
  }
}
```

这是一个**必需**的配置选项。

---

### temperature

使用 `temperature` 配置控制 LLM 响应的随机性和创造力。较低的值使响应更专注和确定，较高的值增加创造力和变异性。

opencode.json

```json
{
  "agent": {
    "plan": {
      "temperature": 0.1
    },
    "creative": {
      "temperature": 0.8
    }
  }
}
```

temperature 值通常从 0.0 到 1.0：

- **0.0-0.2**：非常专注和确定性的响应，适用于代码分析和规划
- **0.3-0.5**：平衡的响应，具有一定的创造力，适用于一般开发任务
- **0.6-1.0**：更多创造性和多样化的响应，适用于头脑风暴和探索

opencode.json

```json
{
  "agent": {
    "analyze": {
      "temperature": 0.1,
      "prompt": "{file:./prompts/analysis.txt}"
    },
    "build": {
      "temperature": 0.3
    },
    "brainstorm": {
      "temperature": 0.7,
      "prompt": "{file:./prompts/creative.txt}"
    }
  }
}
```

如果没有指定 temperature，OpenCode 使用模型特定的默认值；通常大多数模型为 0，Qwen 模型为 0.55。

---

### max steps

控制代理在被迫仅以文本响应之前可以执行的最大代理迭代次数。这允许希望控制成本的用户设置代理操作的限制。

如果未设置，代理将继续迭代，直到模型选择停止或用户中断会话。

opencode.json

```json
{
  "agent": {
    "quick-thinker": {
      "description": "Fast reasoning with limited iterations",
      "prompt": "You are a quick thinker. Solve problems with minimal steps.",
      "steps": 5
    }
  }
}
```

当达到限制时，代理会收到一个特殊系统提示，指示其响应其工作的总结和推荐剩余任务。

**警告**

遗留的 `maxSteps` 字段已弃用。请使用 `steps`。

---

### disable

设置为 `true` 以禁用代理。

opencode.json

```json
{
  "agent": {
    "review": {
      "disable": true
    }
  }
}
```

---

### prompt

使用 `prompt` 配置为代理指定自定义系统提示文件。提示文件应包含特定于代理目的的说明。

opencode.json

```json
{
  "agent": {
    "review": {
      "prompt": "{file:./prompts/code-review.txt}"
    }
  }
}
```

此路径相对于配置文件所在的位置。因此这适用于全局 OpenCode 配置和项目特定配置。

---

### model

使用 `model` 配置覆盖此代理的模型。用于使用针对不同任务优化的不同模型。例如，规划使用更快的模型，实现使用更强大的模型。

**提示**

如果不指定模型，主代理使用[全局配置的模型](./09-配置.md#models)，而子代理将使用调用子代理的主代理的模型。

opencode.json

```json
{
  "agent": {
    "plan": {
      "model": "anthropic/claude-haiku-4-20250514"
    }
  }
}
```

OpenCode 配置中的模型 ID 使用格式 `provider/model-id`。例如，如果你使用 [OpenCode Zen](./07-Zen.md)，对于 GPT 5.1 Codex 你将使用 `opencode/gpt-5.1-codex`。

---

### tools（已弃用）

`tools` **已弃用**。新配置、更新和更精细的控制请首选代理的 [`permission`](#permissions) 字段。

允许你控制此代理中可用的工具。你可以通过将它们设置为 `true` 或 `false` 来启用或禁用特定工具。在代理的 `tools` 配置中，`true` 等效于 `{"*": "allow"}` 权限，`false` 等效于 `{"*": "deny"}` 权限。

opencode.json

```json
{
  "$schema": "https://opencode.ai/config.json",
  "tools": {
    "write": true,
    "bash": true
  },
  "agent": {
    "plan": {
      "tools": {
        "write": false,
        "bash": false
      }
    }
  }
}
```

**注意**

代理特定配置覆盖全局配置。

你也可以在遗留的 `tools` 条目中使用通配符同时控制多个工具。例如，要禁用来自 MCP 服务器的所有工具：

opencode.json

```json
{
  "$schema": "https://opencode.ai/config.json",
  "agent": {
    "readonly": {
      "tools": {
        "mymcp_*": false,
        "write": false,
        "edit": false
      }
    }
  }
}
```

[了解更多关于工具](./10-工具.md)。

---

### permissions

你可以配置权限来管理代理可以执行的操作。目前，`edit`、`bash` 和 `webfetch` 工具的权限可以配置为：

- `"ask"` — 在运行工具之前请求批准
- `"allow"` — 无需批准允许所有操作
- `"deny"` — 禁用工具

opencode.json

```json
{
  "$schema": "https://opencode.ai/config.json",
  "permission": {
    "edit": "deny"
  }
}
```

你可以为每个代理覆盖这些权限。

opencode.json

```json
{
  "$schema": "https://opencode.ai/config.json",
  "permission": {
    "edit": "deny"
  },
  "agent": {
    "build": {
      "permission": {
        "edit": "ask"
      }
    }
  }
}
```

你也可以在 Markdown 代理中设置权限。

~/.config/opencode/agents/review.md

```markdown
---description: Code review without editsmode: subagentpermission:  edit: deny  bash:    "*": ask    "git diff": allow    "git log*": allow    "grep *": allow  webfetch: deny---
Only analyze code and suggest changes.
```

你可以为特定的 bash 命令设置权限。

opencode.json

```json
{
  "$schema": "https://opencode.ai/config.json",
  "agent": {
    "build": {
      "permission": {
        "bash": {
          "git push": "ask",
          "grep *": "allow"
        }
      }
    }
  }
}
```

这可以采用 glob 模式。

opencode.json

```json
{
  "$schema": "https://opencode.ai/config.json",
  "agent": {
    "build": {
      "permission": {
        "bash": {
          "git *": "ask"
        }
      }
    }
  }
}
```

你也可以使用 `*` 通配符来管理所有命令的权限。由于最后一个匹配规则优先，将 `*` 通配符放在第一位，具体规则放在后面。

opencode.json

```json
{
  "$schema": "https://opencode.ai/config.json",
  "agent": {
    "build": {
      "permission": {
        "bash": {
          "*": "ask",
          "git status *": "allow"
        }
      }
    }
  }
}
```

[了解更多关于权限](./10-权限.md)。

---

### mode

使用 `mode` 配置控制代理的模式。`mode` 选项用于确定代理如何被使用。

opencode.json

```json
{
  "agent": {
    "review": {
      "mode": "subagent"
    }
  }
}
```

`mode` 选项可以设置为 `primary`、`subagent` 或 `all`。如果未指定 `mode`，默认为 `all`。

---

### hidden

使用 `hidden: true` 将子代理从 `@` 自动完成菜单中隐藏。适用于仅通过 Task 工具由其他代理以编程方式调用的内部子代理。

opencode.json

```json
{
  "agent": {
    "internal-helper": {
      "mode": "subagent",
      "hidden": true
    }
  }
}
```

这仅影响用户在自动完成菜单中的可见性。如果权限允许，隐藏的代理仍然可以通过 Task 工具由模型调用。

**注意**

仅适用于 `mode: subagent` 代理。

---

### task permissions

使用 `permission.task` 控制代理可以通过 Task 工具调用哪些子代理。使用 glob 模式进行灵活匹配。

opencode.json

```json
{
  "agent": {
    "orchestrator": {
      "mode": "primary",
      "permission": {
        "task": {
          "*": "deny",
          "orchestrator-*": "allow",
          "code-reviewer": "ask"
        }
      }
    }
  }
}
```

当设置为 `deny` 时，子代理从 Task 工具描述中完全移除，因此模型不会尝试调用它。

**提示**

规则按顺序评估，**最后一个匹配规则胜出**。在上面的示例中，`orchestrator-planner` 同时匹配 `*`（deny）和 `orchestrator-*`（allow），但由于 `orchestrator-*` 位于 `*` 之后，结果是 `allow`。

**提示**

用户始终可以通过 `@` 自动完成菜单直接调用任何子代理，即使代理的任务权限会拒绝它。

---

### color

使用 `color` 选项自定义代理在 UI 中的视觉外观。这影响它在界面中的显示方式。

使用有效的十六进制颜色（例如 `#FF5733`）或主题颜色：`primary`、`secondary`、`accent`、`success`、`warning`、`error`、`info`。

opencode.json

```json
{
  "agent": {
    "creative": {
      "color": "#ff6b6b"
    },
    "code-reviewer": {
      "color": "accent"
    }
  }
}
```

---

### top_p

使用 `top_p` 选项控制响应多样性。temperature 的替代方案，用于控制随机性。

opencode.json

```json
{
  "agent": {
    "brainstorm": {
      "top_p": 0.9
    }
  }
}
```

值范围从 0.0 到 1.0。较低的值更专注，较高的值更多样化。

---

### additional

你指定的任何其他选项都将**直接传递**给提供商作为模型选项。这允许你使用提供商特定的功能和参数。

例如，对于 OpenAI 的推理模型，你可以控制推理努力：

opencode.json

```json
{
  "agent": {
    "deep-thinker": {
      "description": "Agent that uses high reasoning effort for complex problems",
      "model": "openai/gpt-5",
      "reasoningEffort": "high",
      "textVerbosity": "low"
    }
  }
}
```

这些额外的选项是特定于模型和提供商的。检查你的提供商的文档以获取可用参数。

**提示**

运行 `opencode models` 查看可用模型列表。

## 创建代理

你可以使用以下命令创建新代理：

终端窗口

```bash
opencode agent create
```

此交互式命令将：

1.  询问保存代理的位置；全局或项目特定。
2.  代理应该做什么的描述。
3.  生成适当的系统提示和标识符。
4.  让你选择代理可以访问哪些工具。
5.  最后，创建一个带有代理配置的 markdown 文件。

---

## 使用场景

以下是不同代理的常见使用场景。

-   **Build 代理**：具有所有工具启用的完整开发工作
-   **Plan 代理**：分析并规划而不进行更改
-   **Review 代理**：具有只读访问权限加上文档工具的代码审查
-   **Debug 代理**：专注于使用启用的 bash 和 read 工具进行调查
-   **Docs 代理**：具有文件操作但没有系统命令的文档编写
